# <a name="state-and-data-in-docker-applications"></a><span data-ttu-id="baa9e-103">État et données dans les applications Docker</span><span class="sxs-lookup"><span data-stu-id="baa9e-103">State and data in Docker applications</span></span>

<span data-ttu-id="baa9e-104">Dans la plupart des cas, vous pouvez considérer qu’un conteneur est une instance d’un processus.</span><span class="sxs-lookup"><span data-stu-id="baa9e-104">In most cases, you can think of a container as an instance of a process.</span></span> <span data-ttu-id="baa9e-105">Un processus ne conserve pas un état persistant.</span><span class="sxs-lookup"><span data-stu-id="baa9e-105">A process does not maintain persistent state.</span></span> <span data-ttu-id="baa9e-106">Si un conteneur peut écrire dans son stockage local, supposer qu’une instance sera présente indéfiniment revient à supposer qu’un seul emplacement en mémoire pourra perdurer.</span><span class="sxs-lookup"><span data-stu-id="baa9e-106">While a container can write to its local storage, assuming that an instance will be around indefinitely is like assuming that a single location in memory will be durable.</span></span> <span data-ttu-id="baa9e-107">Il convient de supposer que les images conteneur, comme les processus, peuvent avoir plusieurs instances et qu’ils seront finalement supprimés ; s’ils sont gérés avec un orchestrateur de conteneurs, il faut supposer qu’ils peuvent être déplacés d’un nœud ou d’une machine virtuelle à l’autre.</span><span class="sxs-lookup"><span data-stu-id="baa9e-107">Container images, like processes, should be assumed to have multiple instances and that they will eventually be killed; if they're managed with a container orchestrator, it should be assumed that they might get moved from one node or VM to another.</span></span>

<span data-ttu-id="baa9e-108">Les solutions suivantes sont utilisées pour gérer les données persistantes dans les applications Docker :</span><span class="sxs-lookup"><span data-stu-id="baa9e-108">The following solutions are used to manage persistent data in Docker applications:</span></span>

<span data-ttu-id="baa9e-109">À partir de l’hôte Docker, sous forme de [volumes Docker](https://docs.docker.com/engine/admin/volumes/) :</span><span class="sxs-lookup"><span data-stu-id="baa9e-109">From the Docker host, as [Docker Volumes](https://docs.docker.com/engine/admin/volumes/):</span></span>

- <span data-ttu-id="baa9e-110">Les **volumes** sont stockés dans une zone du système de fichiers hôte managée par Docker.</span><span class="sxs-lookup"><span data-stu-id="baa9e-110">**Volumes** are stored in an area of the host filesystem that's managed by Docker.</span></span>

- <span data-ttu-id="baa9e-111">Les **montages de liaison** peuvent être mappés à n’importe quel dossier du système de fichiers hôte. <span data-ttu-id="baa9e-113">À partir du stockage distant :</span><span class="sxs-lookup"><span data-stu-id="baa9e-113">From remote storage:</span></span>

- <span data-ttu-id="baa9e-114">Le service [Stockage Azure](https://azure.microsoft.com/documentation/services/storage/) fournit un stockage géographiquement distribuable, offrant une bonne solution de persistance à long terme pour les conteneurs.</span><span class="sxs-lookup"><span data-stu-id="baa9e-114">[Azure Storage](https://azure.microsoft.com/documentation/services/storage/) provides geo-distributable storage, providing a good long-term persistence solution for containers.</span></span>

- <span data-ttu-id="baa9e-115">Des bases de données relationnelles distantes, comme [Azure SQL Database](https://azure.microsoft.com/services/sql-database/), des bases de données NoSQL, comme [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction) ou des services de cache, comme [Redis](https://redis.io/).</span><span class="sxs-lookup"><span data-stu-id="baa9e-115">Remote relational databases like [Azure SQL Database](https://azure.microsoft.com/services/sql-database/), NoSQL databases like [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), or cache services like [Redis](https://redis.io/).</span></span>

<span data-ttu-id="baa9e-116">À partir du conteneur Docker :</span><span class="sxs-lookup"><span data-stu-id="baa9e-116">From the Docker container:</span></span>

- <span data-ttu-id="baa9e-117">Docker fournit une fonctionnalité appelée *système de fichiers par superposition*.</span><span class="sxs-lookup"><span data-stu-id="baa9e-117">Docker provides a feature named the *overlay file system*.</span></span> <span data-ttu-id="baa9e-118">Cette fonctionnalité implémente une tâche de copie sur écriture qui stocke les informations mises à jour dans le système de fichiers racine du conteneur.</span><span class="sxs-lookup"><span data-stu-id="baa9e-118">This feature implements a copy-on-write task that stores updated information to the root file system of the container.</span></span> <span data-ttu-id="baa9e-119">Ces informations « reposent » sur l’image d’origine sur laquelle le conteneur est basé.</span><span class="sxs-lookup"><span data-stu-id="baa9e-119">That information "lays on top of" the original image on which the container is based.</span></span> <span data-ttu-id="baa9e-120">Si le conteneur est supprimé du système, ces modifications sont perdues.</span><span class="sxs-lookup"><span data-stu-id="baa9e-120">If the container is deleted from the system, those changes are lost.</span></span> <span data-ttu-id="baa9e-121">Ainsi, bien qu’il soit possible d’enregistrer l’état d’un conteneur dans son stockage local, la conception d’un système qui s’appuie sur cette fonctionnalité est en conflit avec le principe de conception des conteneurs, lesquels sont par défaut sans état.</span><span class="sxs-lookup"><span data-stu-id="baa9e-121">Therefore, while it's possible to save the state of a container within its local storage, designing a system based on this feature would conflict with the premise of container design, which by default is stateless.</span></span>

<span data-ttu-id="baa9e-124">Ces options sont détaillées ci-après.</span><span class="sxs-lookup"><span data-stu-id="baa9e-124">The following provides additional detail about these options.</span></span>

<span data-ttu-id="baa9e-125">Les **volumes** sont des répertoires mappés du système d’exploitation hôte à des répertoires situés dans les conteneurs.</span><span class="sxs-lookup"><span data-stu-id="baa9e-125">**Volumes** are directories mapped from the host OS to directories in containers.</span></span> <span data-ttu-id="baa9e-126">Quand le code du conteneur a accès au répertoire, cet accès se fait en réalité à un répertoire sur le système d’exploitation hôte.</span><span class="sxs-lookup"><span data-stu-id="baa9e-126">When code in the container has access to the directory, that access is actually to a directory on the host OS.</span></span> <span data-ttu-id="baa9e-127">Ce répertoire n’est pas lié à la durée de vie du conteneur, il est managé par Docker et isolé de la fonctionnalité principale de la machine hôte.</span><span class="sxs-lookup"><span data-stu-id="baa9e-127">This directory is not tied to the lifetime of the container itself, and the directory is managed by Docker and isolated from the core functionality of the host machine.</span></span> <span data-ttu-id="baa9e-128">Ainsi, les volumes de données sont conçus pour rendre des données persistantes indépendamment de la durée de vie du conteneur.</span><span class="sxs-lookup"><span data-stu-id="baa9e-128">Thus, data volumes are designed to persist data independently of the life of the container.</span></span> <span data-ttu-id="baa9e-129">Si vous supprimez un conteneur ou une image de l’hôte Docker, les données rendues persistantes dans le volume de données ne sont pas supprimées.</span><span class="sxs-lookup"><span data-stu-id="baa9e-129">If you delete a container or an image from the Docker host, the data persisted in the data volume is not deleted.</span></span>

<span data-ttu-id="baa9e-130">Les volumes peuvent être nommés ou anonymes (par défaut).</span><span class="sxs-lookup"><span data-stu-id="baa9e-130">Volumes can be named or anonymous (the default).</span></span> <span data-ttu-id="baa9e-131">Les volumes nommés correspondent à l’évolution des **conteneurs de volumes de données**. <span data-ttu-id="baa9e-133">Les **montages de liaison** existent depuis longtemps. Les volumes sont donc l’option recommandée.</span><span class="sxs-lookup"><span data-stu-id="baa9e-134">Bind mounts have more limitations than volumes and some important security issues, so volumes are the recommended option.</span></span>

<span data-ttu-id="baa9e-135">les supports sont des dossiers virtuels qui ne vivent que dans la mémoire de l’hôte et ne sont jamais écrits au système de fichiers. \*\* `tmpfs` \*\*</span><span class="sxs-lookup"><span data-stu-id="baa9e-135">**`tmpfs` mounts** are virtual folders that live only in the host's memory and are never written to the filesystem.</span></span> <span data-ttu-id="baa9e-136">Ils sont rapides et sécurisés, mais ils consomment de la mémoire et sont conçus uniquement pour des données non persistantes.</span><span class="sxs-lookup"><span data-stu-id="baa9e-136">They are fast and secure but use memory and are only meant for non-persistent data.</span></span>

<span data-ttu-id="baa9e-137">Comme le montre la figure 4-5, les volumes Docker standard peuvent être stockés en dehors des conteneurs eux-mêmes, mais dans les limites physiques du serveur ou de la machine virtuelle hôte.</span><span class="sxs-lookup"><span data-stu-id="baa9e-137">As shown in Figure 4-5, regular Docker volumes can be stored outside of the containers themselves but within the physical boundaries of the host server or VM.</span></span> <span data-ttu-id="baa9e-138">Les conteneurs Docker ne peuvent cependant pas accéder à un volume depuis un serveur ou une machine virtuelle hôte à un autre.</span><span class="sxs-lookup"><span data-stu-id="baa9e-138">However, Docker containers cannot access a volume from one host server or VM to another.</span></span> <span data-ttu-id="baa9e-139">En d’autres termes, avec ces volumes, il n’est pas possible de gérer les données partagées entre des conteneurs qui s’exécutent sur des hôtes Docker distincts.
