# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="1c157-102">Abstractions (Types et interfaces abstraits)</span><span class="sxs-lookup"><span data-stu-id="1c157-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="1c157-103">Une abstraction est un type qui décrit un contrat, mais qui ne fournit pas une implémentation complète du contrat.</span><span class="sxs-lookup"><span data-stu-id="1c157-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="1c157-104">Les abstractions sont généralement implémentées en tant que classes abstraites ou interfaces, et elles sont fournies avec un ensemble bien défini de documentation de référence décrivant la sémantique requise des types qui implémentent le contrat.</span><span class="sxs-lookup"><span data-stu-id="1c157-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="1c157-105">Voici quelques-unes des abstractions les plus importantes du .NET Framework : <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>et <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="1c157-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>

 <span data-ttu-id="1c157-106">Vous pouvez étendre des frameworks en implémentant un type concret qui prend en charge le contrat d’une abstraction et à l’aide de ce type concret avec les API d’infrastructure consommant (fonctionnant sur) l’abstraction.</span><span class="sxs-lookup"><span data-stu-id="1c157-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>

 <span data-ttu-id="1c157-107">Une abstraction significative et utile qui peut résister au test de temps est très difficile à concevoir.</span><span class="sxs-lookup"><span data-stu-id="1c157-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="1c157-108">La difficulté principale est l’obtention du jeu de membres approprié, et pas plus ou moins.</span><span class="sxs-lookup"><span data-stu-id="1c157-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="1c157-109">Si une abstraction a trop de membres, elle devient difficile, voire impossible à implémenter.</span><span class="sxs-lookup"><span data-stu-id="1c157-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="1c157-110">S’il a trop peu de membres pour la fonctionnalité promis, il devient inutile dans de nombreux scénarios intéressants.</span><span class="sxs-lookup"><span data-stu-id="1c157-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>

 <span data-ttu-id="1c157-111">Un trop grand nombre d’abstractions dans une infrastructure affecte également la convivialité de l’infrastructure.</span><span class="sxs-lookup"><span data-stu-id="1c157-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="1c157-112">Il est souvent assez difficile de comprendre une abstraction sans comprendre comment elle s’intègre à la plus grande image des implémentations concrètes et aux API opérant sur l’abstraction.</span><span class="sxs-lookup"><span data-stu-id="1c157-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="1c157-113">En outre, les noms des abstractions et de leurs membres sont nécessairement abstraits, ce qui les rend souvent énigmatiques et incompréhensibles sans avoir d’abord à comprendre le contexte plus large de leur utilisation.</span><span class="sxs-lookup"><span data-stu-id="1c157-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>

 <span data-ttu-id="1c157-114">Toutefois, les abstractions offrent une extensibilité extrêmement puissante que les autres mécanismes d’extensibilité ne correspondent pas souvent.</span><span class="sxs-lookup"><span data-stu-id="1c157-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="1c157-115">Ils sont au cœur de nombreux modèles architecturaux, tels que les plug-ins, l’inversion de contrôle (IoC), les pipelines, etc.</span><span class="sxs-lookup"><span data-stu-id="1c157-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="1c157-116">Ils sont également extrêmement importants pour la testabilité des frameworks.</span><span class="sxs-lookup"><span data-stu-id="1c157-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="1c157-117">De bonnes abstractions permettent de remplacer les dépendances lourdes dans le cadre des tests unitaires.</span><span class="sxs-lookup"><span data-stu-id="1c157-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="1c157-118">En résumé, les abstractions sont responsables de la richesse recherchée des frameworks orientés objet modernes.</span><span class="sxs-lookup"><span data-stu-id="1c157-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>

 <span data-ttu-id="1c157-119">❌ ne fournissez pas d’abstractions, à moins qu’elles ne soient testées en développant plusieurs implémentations concrètes et API consommant les abstractions.</span><span class="sxs-lookup"><span data-stu-id="1c157-119">❌ DO NOT provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>

 <span data-ttu-id="1c157-120">✔️ choisir soigneusement entre une classe abstraite et une interface lors de la conception d’une abstraction.</span><span class="sxs-lookup"><span data-stu-id="1c157-120">✔️ DO choose carefully between an abstract class and an interface when designing an abstraction.</span></span>

 <span data-ttu-id="1c157-121">✔️ envisagez de fournir des tests de référence pour les implémentations concrètes des abstractions.</span><span class="sxs-lookup"><span data-stu-id="1c157-121">✔️ CONSIDER providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="1c157-122">Ces tests doivent permettre aux utilisateurs de tester si leurs implémentations implémentent correctement le contrat.</span><span class="sxs-lookup"><span data-stu-id="1c157-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
