## <a name="remarks"></a><span data-ttu-id="63824-106">Notes</span><span class="sxs-lookup"><span data-stu-id="63824-106">Remarks</span></span>

<span data-ttu-id="63824-107">Pour chaque fichier de code F#, vous pouvez avoir un *fichier de signature*, qui est un fichier portant le même nom que le fichier de code, mais avec l’extension .fsi au lieu de .fs.</span><span class="sxs-lookup"><span data-stu-id="63824-107">For each F# code file, you can have a *signature file*, which is a file that has the same name as the code file but with the extension .fsi instead of .fs.</span></span> <span data-ttu-id="63824-108">Les fichiers de signature peuvent également être ajoutés à la ligne de commande de compilation si vous utilisez la ligne de commande directement.</span><span class="sxs-lookup"><span data-stu-id="63824-108">Signature files can also be added to the compilation command-line if you are using the command line directly.</span></span> <span data-ttu-id="63824-109">Pour faire la distinction entre les fichiers de code et les fichiers de signature, les fichiers de code sont parfois appelés *fichiers d’implémentation*.</span><span class="sxs-lookup"><span data-stu-id="63824-109">To distinguish between code files and signature files, code files are sometimes referred to as *implementation files*.</span></span> <span data-ttu-id="63824-110">Dans un projet, le fichier de signature doit précéder le fichier de code associé.</span><span class="sxs-lookup"><span data-stu-id="63824-110">In a project, the signature file should precede the associated code file.</span></span>

<span data-ttu-id="63824-117">Dans un fichier de signature, vous ne répétez pas la définition des types et les implémentations de chaque méthode ou fonction.</span><span class="sxs-lookup"><span data-stu-id="63824-117">In a signature file, you do not repeat the definition of the types and the implementations of each method or function.</span></span> <span data-ttu-id="63824-118">Au lieu de cela, vous utilisez la signature pour chaque méthode et fonction, qui sert de spécification complète des fonctionnalités implémentées par un fragment de module ou d’espace de noms.</span><span class="sxs-lookup"><span data-stu-id="63824-118">Instead, you use the signature for each method and function, which acts as a complete specification of the functionality that is implemented by a module or namespace fragment.</span></span> <span data-ttu-id="63824-119">La syntaxe pour une signature de type est la même que celle utilisée dans les déclarations de méthode abstraites dans les interfaces et les classes abstraites, et est également indiquée par IntelliSense et l’interpréteur F# fsi.exe lorsqu’il affiche correctement l’entrée compilée.</span><span class="sxs-lookup"><span data-stu-id="63824-119">The syntax for a type signature is the same as that used in abstract method declarations in interfaces and abstract classes, and is also shown by IntelliSense and by the F# interpreter fsi.exe when it displays correctly compiled input.</span></span>

<span data-ttu-id="63824-120">Quand il n’y a pas assez d’informations dans la signature de type pour indiquer si un type est sealed ou non, ou s’il s’agit d’un type d’interface, vous devez ajouter un attribut qui indique la nature du type au compilateur.</span><span class="sxs-lookup"><span data-stu-id="63824-120">If there is not enough information in the type signature to indicate whether a type is sealed, or whether it is an interface type, you must add an attribute that indicates the nature of the type to the compiler.</span></span> <span data-ttu-id="63824-121">Les attributs utilisés à cette fin sont décrits dans le tableau suivant.</span><span class="sxs-lookup"><span data-stu-id="63824-121">Attributes that you use for this purpose are described in the following table.</span></span>

|<span data-ttu-id="63824-122">Attribut</span><span class="sxs-lookup"><span data-stu-id="63824-122">Attribute</span></span>|<span data-ttu-id="63824-123">Description</span><span class="sxs-lookup"><span data-stu-id="63824-123">Description</span></span>|
|---------|-----------|
|`[<Sealed>]`|<span data-ttu-id="63824-124">Type qui n’a pas de membres abstraits ou qui ne doit pas être étendu.</span><span class="sxs-lookup"><span data-stu-id="63824-124">For a type that has no abstract members, or that should not be extended.</span></span>|
|`[<Interface>]`|<span data-ttu-id="63824-125">Type qui est une interface.</span><span class="sxs-lookup"><span data-stu-id="63824-125">For a type that is an interface.</span></span>|

<span data-ttu-id="63824-127">Utilisez le mot clé `val` pour créer une signature pour une valeur ou une valeur de fonction.</span><span class="sxs-lookup"><span data-stu-id="63824-127">Use the keyword `val` to create a signature for a value or function value.</span></span> <span data-ttu-id="63824-128">Le mot clé `type` introduit une signature de type.</span><span class="sxs-lookup"><span data-stu-id="63824-128">The keyword `type` introduces a type signature.</span></span>

<span data-ttu-id="63824-129">Vous pouvez générer un fichier de signature à l’aide de l’option de compilateur `--sig` .</span><span class="sxs-lookup"><span data-stu-id="63824-129">You can generate a signature file by using the `--sig` compiler option.</span></span> <span data-ttu-id="63824-130">En général, vous n’écrivez pas les fichiers .fsi manuellement.</span><span class="sxs-lookup"><span data-stu-id="63824-130">Generally, you do not write .fsi files manually.</span></span> <span data-ttu-id="63824-131">À la place, vous générez des fichiers .fsi à l’aide du compilateur, vous les ajoutez à votre projet, si vous en avez un, et vous les modifiez en supprimant les méthodes et les fonctions que vous ne souhaitez pas rendre accessibles.</span><span class="sxs-lookup"><span data-stu-id="63824-131">Instead, you generate .fsi files by using the compiler, add them to your project, if you have one, and edit them by removing methods and functions that you do not want to be accessible.</span></span>

<span data-ttu-id="63824-132">Les signatures de type sont soumises à plusieurs règles :</span><span class="sxs-lookup"><span data-stu-id="63824-132">There are several rules for type signatures:</span></span>

- <span data-ttu-id="63824-134">Les enregistrements et les unions discriminées doivent exposer la totalité ou aucun de leurs champs et constructeurs, et l’ordre dans la signature doit correspondre à celui dans le fichier d’implémentation.</span><span class="sxs-lookup"><span data-stu-id="63824-134">Records and discriminated unions must expose either all or none of their fields and constructors, and the order in the signature must match the order in the implementation file.</span></span> <span data-ttu-id="63824-135">Les classes peuvent révéler une partie, la totalité ou aucun des champs et des méthodes dans la signature.</span><span class="sxs-lookup"><span data-stu-id="63824-135">Classes can reveal some, all, or none of their fields and methods in the signature.</span></span>

- <span data-ttu-id="63824-136">Les classes et les structures qui ont des constructeurs doivent exposer les déclarations de leurs classes de base (déclaration `inherits` ).</span><span class="sxs-lookup"><span data-stu-id="63824-136">Classes and structures that have constructors must expose the declarations of their base classes (the `inherits` declaration).</span></span> <span data-ttu-id="63824-137">En outre, les classes et les structures qui ont des constructeurs doivent exposer toutes leurs méthodes abstraites et leurs déclarations d’interface.</span><span class="sxs-lookup"><span data-stu-id="63824-137">Also, classes and structures that have constructors must expose all of their abstract methods and interface declarations.</span></span>

- <span data-ttu-id="63824-138">Les types d’interface doivent révéler toutes leurs méthodes et interfaces.</span><span class="sxs-lookup"><span data-stu-id="63824-138">Interface types must reveal all their methods and interfaces.</span></span>

<span data-ttu-id="63824-139">Les signatures de valeur obéissent aux règles suivantes :</span><span class="sxs-lookup"><span data-stu-id="63824-139">The rules for value signatures are as follows:</span></span>

- <span data-ttu-id="63824-141">Le nombre de paramètres de type générique (déduits implicitement ou déclarés explicitement) doit correspondre, et les types et contraintes de type dans les paramètres de type générique doivent correspondre.</span><span class="sxs-lookup"><span data-stu-id="63824-141">The number of generic type parameters (either implicitly inferred or explicitly declared) must match, and the types and type constraints in generic type parameters must match.</span></span>

- <span data-ttu-id="63824-143">Le modèle de paramètres (aussi appelé *arité*) des signatures et des implémentations doit être cohérent.</span><span class="sxs-lookup"><span data-stu-id="63824-143">The pattern of parameters (also known as the *arity*) of signatures and implementations must be consistent.</span></span>

- <span data-ttu-id="63824-144">Si les noms de paramètres dans un fichier de signature diffèrent du fichier d’implémentation correspondant, le nom figurant dans le fichier de signature est utilisé à la place, ce qui peut provoquer des problèmes lors du débogage ou du profilage.</span><span class="sxs-lookup"><span data-stu-id="63824-144">If parameter names in a signature file differ from the corresponding implementation file, the name in the signature file will be used instead, which may cause issues when debugging or profiling.</span></span> <span data-ttu-id="63824-145">Si vous souhaitez être informé de ces incompatibilités, activez l’avertissement 3218 dans votre fichier projet ou lors de l’appel du compilateur ( `--warnon` consultez sous [Options du compilateur](compiler-options.md)).</span><span class="sxs-lookup"><span data-stu-id="63824-145">If you wish to be notified of such mismatches, enable warning 3218 in your project file or when invoking the compiler (see `--warnon` under [Compiler Options](compiler-options.md)).</span></span>
